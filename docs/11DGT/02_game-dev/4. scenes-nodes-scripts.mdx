# Scenes, Nodes, and Scripts

Godot games are built using **scenes**, **nodes**, and **scripts**.

Understanding how these fit together is essential for:
- building mechanics
- organising projects
- explaining how your game works

---

## Nodes

A node is a **single object** in a game.

Examples of nodes include:
- a player
- an enemy
- a platform
- a button
- a sound effect

![Nodes - example game objects such as a player, enemy, platform, button, and sound effect](/img/11DGT/game-dev/nodes-in-godot.svg)

Each node has a specific role.

---

## Scenes

A scene is a **collection of nodes** arranged together.

Scenes are used to:
- build levels
- group related objects
- reuse parts of your game

A scene might represent:
- a level
- a menu
- a player character
- an enemy type

![Scenes - grouping related nodes into levels, menus, characters, and reusable parts](/img/11DGT/game-dev/scenes-in-godot.svg)

## Scripts

Scripts define **behaviour**.

A script:
- is attached to a node
- controls how that node behaves
- responds to input and events

Scripts are written in **GDScript**.

![Scripts - behaviour attached to a node through GDScript](/img/11DGT/game-dev/scripts-in-godot.svg)

## How They Work Together

Scenes, nodes, and scripts work as a system:

- nodes represent objects  
- scripts define behaviour  
- scenes organise objects  

![How They Work Together - a scene contains nodes and each node can have its own script](/img/11DGT/game-dev/how-scenes-nodes-scripts-work-together.svg)

## Why This Structure Matters

Using scenes and nodes correctly:
- reduces complexity
- avoids duplicated code
- makes testing easier
- improves readability

![Why This Structure Matters - reducing complexity, avoiding duplication, easing testing, and improving readability](/img/11DGT/game-dev/why-structure-matters-in-godot.svg)

Poor organisation leads to confusion and bugs.
